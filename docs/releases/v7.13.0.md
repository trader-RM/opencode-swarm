# v7.13.0 — Language-Agnostic Plugin

## What changed

Seven phases of work to lift opencode-swarm out of its TypeScript-first hardcoding and onto a registry-driven `LanguageBackend` extension seam. New languages now ship as a single profile entry plus an optional 60-line backend file.

### Phase 0 — `isCommandAvailable` Invariant 3 fix
- `src/build/discovery.ts:isCommandAvailable` now sets explicit `cwd`, `stdin: 'ignore'`, `timeout: 3000ms`, and `'ignore'` stdio. Pre-fix it omitted three of the four invariant-3 properties; every caller (13 production sites) inherited the violation.
- DI seam (`_internals.spawnSyncImpl`) added so the new `tests/unit/build/discovery.test.ts` "Invariant 3 — subprocess properties" suite asserts the contract directly via spy, not just by reading source.

### Phase 1 — `LanguageProfile` enrichment + drift guards
- `LanguageProfile.treeSitter.commentNodes` added (optional; populated for all 12 production profiles using values from `src/lang/registry.ts`). Source of truth lives in profiles; parser registry is the projection.
- `LanguageRegistry.register()` now THROWS on duplicate id and on duplicate non-`parserOnly` extension claim (was silent overwrite — hid configuration drift).
- `LanguageRegistry.unregister(id)` added so tests that mutate the global singleton can clean up after themselves (Bun's per-file-but-shared-process runner otherwise leaks fixture profiles across test files).
- New `tests/unit/lang/profile-registry-parity.test.ts`: every shared id agrees on `commentNodes`; the asymmetry list (registry-only: `javascript, c, tsx, css, bash, powershell, ini, regex`) is asserted exactly. Adding a language drift-trips this test until both registries are updated.

### Phase 2 — `LanguageBackend` interface, default backend, TypeScript backend
- New `src/lang/backend.ts` interface with optional behavior hooks (`detectProject`, `selectTestFramework`, `buildTestCommand`, `parseTestOutput`, `testFilesFor`, `extractImports`, `selectBuildCommand`).
- `src/lang/default-backend.ts` provides registry-driven defaults for every hook: highest-priority framework whose detect file exists AND whose binary is on PATH (`isCommandAvailable`).
- `src/lang/dispatch.ts` adds `pickBackend(dir)`. Walks up to the manifest root, picks the dominant detected language, returns the registered or defaulted backend. Bounded LRU cache (capacity 64) keyed by manifest hash (`size:mtimeMs:ino`).
- `src/lang/backends/typescript.ts` is the first concrete override. Honors `package.json#scripts.test` first, then `devDependencies` (parity with `test-runner.ts:309–312`), then default. `extractImports` covers ES, BARE, REQUIRE, DYNAMIC, REEXPORT — superset of `analyzer.ts`.
- `tests/unit/lang/backend-purity.test.ts`: static-analysis test failing PRs that import `bun:...`, reference `Bun.*`, or use spawn primitives in any backend file. Backends never spawn — the single spawn site stays in `src/tools/test-runner.ts` and `src/build/discovery.ts:isCommandAvailable`.

### Phase 3 — test-runner dispatch with kill switch
- `src/tools/test-runner.ts` adds `detectTestFrameworkViaDispatch(cwd)`. Gated by `SWARM_LANG_BACKEND=dispatch` env var; default remains the legacy switch. `SWARM_LANG_BACKEND=legacy` becomes the rollback path in a future flip-default change.
- New `tests/unit/tools/test-runner-dispatch-parity.test.ts` asserts weak parity: when both paths return non-`'none'`, they agree on framework identity; the dispatch may be stricter (binary-on-PATH check) than the legacy (manifest content scan).
- `buildTestCommand` and `parseTestOutput` switches deferred to a follow-up — replacing 14-framework switches in one PR is high-risk; keeping that work bisectable.

### Phase 4b — architect prompt template substitution
- `src/agents/template.ts`: strict `{{KEY}}` renderer. Throws on unknown placeholders; auto-escapes backticks + `${...}` for safe inclusion in TS template literals.
- `src/agents/project-context.ts`: `buildProjectContext(directory)` resolves `pickBackend` and queries `selectBuildCommand`, `selectTestFramework`, `selectLintCommand`. Bulleted, escaped per-language constraint blocks ready for `{{CODER_CONSTRAINTS}}` / `{{REVIEWER_CHECKLIST}}` placeholders.
- `src/index.ts:initializeOpenCodeSwarm` calls `buildProjectContext` between `ensureSwarmGitExcluded` and `getAgentConfigs`, wrapped in `withTimeout(2000ms)` with `.catch(() => null)` fail-open per Invariant 1. The architect's existing `{{PROJECT_LANGUAGE}}`, `{{BUILD_CMD}}`, `{{TEST_CMD}}`, `{{LINT_CMD}}` placeholders (which previously shipped as literals) now resolve to the detected backend's values.
- `tests/unit/agents/init-fail-open.test.ts` is the regression guard for v7.0.3 issue #704 + v7.3.3 git-hygiene precedent: a hung `pickBackend` cannot block the manifest return.

### Phase 5 — Python and Go backends
- `src/lang/backends/python.ts`: `import x`, `import x.y`, `import x as y`, `import a, b`, `from x import y` (relative `.`/`.foo` and dotted `x.y`), multi-line `from x import (\n a,\n b\n)`. Aliases keep the source module per analyzer convention.
- `src/lang/backends/go.ts`: single-line `import "x"`, aliased / side-effect / dot variants, and grouped `import (\n "a"\n "b"\n)` with mixed alias forms inside.
- 20 new tests in `tests/unit/lang/python-go-backends.test.ts` lock in every supported import shape per language plus dedup and registration.
- Routing the test-impact analyzer (`src/test-impact/analyzer.ts`) through these backends for non-TS files is deferred — the analyzer's `findTestFilesSync` filters to TS extensions today and expanding it requires extensive parity testing.

### Phase 6 — extension-seam documentation
- `docs/adding-a-language.md`: contributor guide covering the three-registry architecture, step-by-step profile/parser/backend additions, backend invariants, common pitfalls, rollout checklist.
- README.md and `docs/index.md` updated with cross-references.

## Why

Three motivations:

1. **Project-context substitution** — the architect prompt has shipped with literal `{{PROJECT_LANGUAGE}}` / `{{BUILD_CMD}}` placeholders for months. The model receives unrendered curly braces, which is both confusing and a wasted opportunity to ground the agent in the user's actual project. Phase 4b closes that.
2. **Invariant 3 hygiene** — `isCommandAvailable` was a known anti-pattern (no cwd, no stdin, no timeout) that several past regressions traced back to. Phase 0 closes it as a prerequisite for the dispatch layer's binary-availability checks.
3. **Extensibility** — adding a language used to mean editing `src/tools/test-runner.ts:1858` (a 14-way switch on framework detection), `src/agents/coder.ts:125`, `src/agents/test-engineer.ts:67`, and several hooks. Now it's a single profile entry and an optional backend file.

## Migration

No action required for end users. The architect prompt now substitutes its placeholders with detected values; if detection fails (no manifest, slow filesystem, etc.), the placeholders resolve to the literal sentinel `unresolved (run /swarm preflight)` — the architect's existing DISCOVER mode picks this up the same way it always has.

For plugin developers extending opencode-swarm to a new language, see [`docs/adding-a-language.md`](../adding-a-language.md).

The dispatch path is opt-in via `SWARM_LANG_BACKEND=dispatch` for one release. Behavior is unchanged in the default (legacy) path. A future minor version will flip the default and `SWARM_LANG_BACKEND=legacy` will become the rollback for one further release.

## Breaking changes

None. Every change is additive to public APIs:

- `getAgentConfigs(config?, directory?, sessionId?, projectContext?)` adds an optional 4th argument; existing 3-arg callers unaffected.
- `createAgents(config?, projectContext?)` adds an optional 2nd argument.
- `LanguageProfile.treeSitter.commentNodes?: string[]` is optional; existing test fixtures and externally-defined profiles type-check.
- `LanguageRegistry.register()` now throws on duplicate ids and extension collisions (was silent overwrite). Three internal tier1/tier2 tests that asserted the old silent-overwrite behavior were updated to assert the new throw.
- `SWARM_LANG_BACKEND=dispatch` opt-in for the new test-runner detection path; default (`unset`) is the legacy switch.

## Known caveats

- **Tier-2 framework name dispatch** — Phase 3 only routes `detectTestFramework` through dispatch. `buildTestCommand` (the 14-framework switch at `test-runner.ts:~894-1042`) and `parseTestOutput` (`~1046-1284`) still run from the legacy switch even when `SWARM_LANG_BACKEND=dispatch` is set. They'll be lifted in a follow-up.
- **Python/Go in the impact graph** — Phase 5 ships the import extractors but `src/test-impact/analyzer.ts` still scans only TS files. Adding non-TS to the graph is a separate change.
- **Heuristic divergence on Rust/Python** — when both detection paths (legacy switch and dispatch backend) return non-`'none'`, they agree on framework identity. They CAN disagree on availability:
  - Legacy is more permissive for Python (trusts pyproject.toml content) and stricter for Rust (requires `[dev-dependencies]` block content).
  - Dispatch is more permissive for Rust (any Cargo.toml + cargo on PATH) and stricter for Python (requires pytest binary on PATH).
  - Both behaviors are reasonable; convergence is a Phase 3b heuristic-lift task.
- **Pre-existing test failures** — ~22 unit tests fail per-file isolation on origin/main as well as on this branch. None are caused by this work; sampled six on the main HEAD (b188827) and confirmed identical failure counts. They are noted in the PR body's `## Pre-existing failures` section, not blocked on by this release.
