# v7.4.4 — Security hardening for repo-graph workspace validation

## What changed

### realpathSync fallback security hardening (src/hooks/repo-graph-builder.ts)

The workspace boundary validation in `realpathSync` now rejects file paths where the call fails for reasons other than ENOENT (file not found). Previously, any `realpathSync` failure would silently fall back to the unresolved path, creating a symlink-based workspace escape vulnerability.

**Specific fixes:**
- **EACCES rejection**: Paths that fail with EACCES (permission denied) on symlinks pointing outside the workspace are now rejected. Previously, an attacker could create a symlink with restricted permissions pointing outside the workspace to bypass boundary checks.
- **ELOOP rejection**: Symlink cycles that cause `realpathSync` to fail with ELOOP are now rejected. Previously, these would fall back to unresolved paths, potentially allowing directory traversal.
- **Other errors rejected**: Any `realpathSync` failure except ENOENT now causes the path to be rejected. Only ENOENT triggers the fallback (for new files being created that don't exist yet).

### Dead code removal

- **Removed redundant barrel file**: Deleted `src/tools/repo-graph/index.ts`, a redundant re-export barrel that duplicated the canonical public entry point at `src/tools/repo-graph.ts`. Zero consumers imported from the deleted path.

## Why

The original `realpathSync` fallback logic treated all errors identically — any failure would fall back to the unresolved path. This created a security vulnerability where an attacker could bypass workspace boundary checks via:

1. **Symlink escape**: Creating a symlink with restricted permissions (EACCES) pointing outside the workspace, causing `realpathSync` to fail and fall back to the malicious path.
2. **Symlink cycles**: Creating circular symlinks (ELOOP) that cause `realpathSync` to fail, again falling back to an unvalidated path.

By only allowing ENOENT to trigger the fallback (which is acceptable for new files being created), all other errors now cause the path to be rejected, maintaining the workspace boundary invariant.

## Migration steps

No migration required. These are security hardening changes that do not alter the public API or behavior for legitimate use cases.
